Featured research

Flagship project
Are SHAP & LIME Faithful on Tabular Foundation Models?
TabPFN · KernelSHAP · LIME · Infidelity · Spearman ρ · Privacy Analysis · NTU
SHAP ρ = 0.474 LIME ρ = −0.060 (noise) Intrinsic probe ρ = −0.350 TabPFN acc 90% vs XGB 86.7%
Problem
SHAP and LIME assume fixed model weights. TabPFN uses in-context learning — the training set is the prompt. When SHAP masks a feature, attention retrieves different context neighbours, shifting what task the model solves. You end up measuring feature importance + context drift mixed together, inseparably.
Approach
Measured three faithfulness metrics — Infidelity (Yeh et al., 2019), Sufficiency, and Spearman ρ against XGBoost-TreeSHAP as exact Shapley reference — across KernelSHAP, LIME, and a finite-difference sensitivity probe on the Adult Income dataset (N=100 train, 30 test, 14 features).
Result
SHAP is locally faithful (ρ=0.474) but LIME is noise (ρ=−0.06). Most strikingly, the intrinsic probe is negatively correlated (ρ=−0.35) — TabPFN and XGBoost learned genuinely different decision boundaries from identical data. Extended with a privacy risk analysis: ICL training data is in the prompt, creating membership inference attack surfaces classical ML doesn't have.
Insight
Post-hoc tools can be locally consistent and still miss the model's actual reasoning. Privacy risk analysis maps ICL vulnerabilities to PDPA obligations (Sections 18, 24, 26) and NIST AI RMF controls — directly applicable to enterprise AI governance and Privacy Impact Assessments.

Fully autonomous maze navigation — bare-metal STM32 firmware
NTU MDP Autonomous Robot
↗
Sub-cm wall alignment 9 FreeRTOS tasks Heading error <1° on 90° turns OpenCV obstacle detection
Problem
Navigate a 10×10 maze autonomously, identify all image targets, compute the fastest return path, execute without any human input.
Approach
Wrote the entire STM32 firmware: 9 concurrent FreeRTOS tasks fusing ICM20948 IMU, dual 480-CPR encoders, HC-SR04 ultrasonic and Sharp IR. EMI-protected I2C, log-log calibrated IR, anti-windup PID, DMA-buffered UART bridge to Raspberry Pi. Added Python OpenCV obstacle detection + Flask REST API.
Result
Reliable navigation over repeated runs with sub-centimetre wall alignment. Gyro-corrected turns held heading error under 1° across 90° manoeuvres.
What I learned
80% of autonomous reliability is firmware, not algorithm. Interrupt timing, DMA buffer sizing and task priority tuning mattered more than the path planning algorithm running upstairs on the Pi.
State estimation + sensor health monitoring for autonomous robots
EKF Sensor Fusion + Signal Intelligence
↗
2.04 cm RMSE 65.7% better than dead reckoning 4 fault types detected Welch PSD · SNR 38 dB
Problem
Dead reckoning drifts to 6 cm error over 27 seconds — each turn compounds heading error into position error. Sensors also fail silently: bias drift, EMI spikes, flatlines.
Approach
EKF with arc-based differential-drive kinematics, 100 Hz gyro correction, 20 Hz ultrasonic updates with 3σ outlier gating. Added online anomaly time-series anomaly detector (spike/drift/flatline/noise burst) using rolling z-score and local drift comparison. Welch PSD spectral analysis with Hanning/Blackman/rectangular window comparison.
Result
2.04 cm RMSE vs 5.96 cm dead reckoning. Anomaly detector catches injected faults with recall up to 0.63 on gyro channel. Spectral analysis reveals 0.31 Hz dominant frequency — gyro SNR 38.4 dB.
What I learned
Heading correction at 100 Hz matters more than ultrasonic updates — fixing the heading stops position errors from ever accumulating. The same Welch PSD pipeline applies directly to physiological and gravimeter sensor signals.
Time-series classification — on-device HAR, 42k params, fits on Arduino
Edge ML — Time-Series Classification · CNN · BiLSTM · GRU
↗
CNN 96.5% · GRU 95.8% 258-dim MFCC features Conditional LSTM-GAN 42k params · ~165 KB az channel importance 0.270
Problem
Classify 6 activities from raw 6-axis IMU windows in real time, on a microcontroller with 256 KB SRAM. Need both accuracy and efficiency.
Approach
MFCC-inspired filterbank pipeline (Hanning → FFT → 12-band filterbank → DCT): Hanning window → FFT → 12-band linear filterbank → log energy + delta + DCT cepstral coefficients = 258-dim feature vector. Three deep architectures + conditional LSTM-GAN for augmentation: CNN1D (42k), BiLSTM (99k), GRU (74k). Feature importance heatmap shows az (vertical accel) dominates at importance 0.270.
Result
CNN1D wins on efficiency: 96.5% at 42k params vs BiLSTM 95.2% at 99k. Conditional LSTM-GAN augmentation: +3.1% gain at 3% training data. CNN fits on-device after int8 quantisation (~11 KB). Walking (2 Hz) and running (3 Hz) separated cleanly by filterbank.
What I learned
MFCC was designed for speech but transfers directly to IMU data — both are fixed-rate signals with energy in frequency bands. Global Average Pooling halves parameter count vs Flatten with no accuracy loss, forcing conv layers to learn genuinely useful temporal features.
Production RAG with OWASP LLM Top 10 #1 defence
Agentic RAG Pipeline with Security Hardening
↗
MRR 1.000 Injection F1 0.909 Precision 1.000 · 0 false positives ReAct · BM25 · Flask API
Problem
Prompt injection (OWASP LLM #1) affects most RAG systems: attackers embed instructions in queries or retrieved documents. Most pipelines have no defence layer at all.
Approach
End-to-end data pipeline: BM25 retrieval (3 chunking strategies) + ReAct agentic loop with tool use (retrieve, calculate, summarise) + 3-layer injection defence: 20+ regex patterns → heuristic scoring (instruction density, entropy) → indirect chunk screening. Deployed via Flask REST API.
Result
Retrieval MRR=1.000 across all strategies. Injection detector: Precision=1.000, Recall=0.833, F1=0.909 — zero false positives, blocks all high-confidence attacks. LLM generation via Claude / GPT-4 API. Connect any LLM via a single callback.
What I learned
Indirect injection (malicious content in retrieved documents) is harder to catch than direct injection — it requires screening every chunk before it enters the context window, not just the user's input. The defence layer must be part of the retrieval pipeline, not bolted on after.
Privacy-preserving ML — quantifying the privacy-utility tradeoff
Federated Learning + Differential Privacy
↗
MIA AUC 0.543→0.510 ε=1.0 DP guarantee FedAvg · 5 clients PDPA + NIST AI RMF
Problem
Classical ML leaks training data through membership inference attacks. Standard DP-SGD doesn't apply to federated settings — the training set is distributed across private clients.
Approach
FedAvg across 5 virtual clients with (ε, δ)-DP via Gaussian mechanism: clip local updates to L2 norm C, add calibrated noise σ = C·sqrt(2·ln(1.25/δ))/ε before aggregation. Shadow-model membership inference attack to quantify leakage.
Result
DP-FedAvg reduces membership inference AUC from 0.543 → 0.510 (near-random = near-perfect privacy resistance) at ε=1.0. Privacy-utility cost explicitly quantified: 37.9% accuracy drop — a concrete number for Privacy Impact Assessments.
What I learned
FL without DP is actually more vulnerable than centralised training — clients overfit to local shards, increasing membership inference risk. DP noise is essential, not optional, in federated deployments. Mapped to PDPA Sections 18, 24, 26 and NIST AI RMF controls.
